# 自定义组合规则

`custom_rules` 是 `config/sources.yaml` 中的一个特殊段落，让你可以**从已有的规则源中挑选几个，合并成一个新的规则文件**。

## 为什么需要自定义组合规则？

默认情况下，FilterFusion 输出两个核心规则文件：

- `adblock-mo.txt` — 手机端规则（m1 + m2 + m3 + b1 + b2）
- `adblock-pc.txt` — 电脑端规则（p1 + b1 + b2）

但你可能想要：

- 一个**更精简的手机端规则**（只含 m1 + b1）
- 一个**实验性规则**（含 m1 + m2 + 某个新源）
- 一个**给家人的规则**（去掉某些过于激进的规则）

`custom_rules` 就是为这些场景设计的。

## 基本用法

### 语法

```yaml
custom_rules:
  - output: 文件名.txt          # 输出到 dist/ 下的文件名
    description: 描述文本        # 可选，写入规则文件头部
    sources: [id1, id2, id3]  # 要合并的源 id 列表
```

### 示例 1：精简手机端规则

```yaml
custom_rules:
  - output: adblock-mo-lite.txt
    description: FilterFusion - Lite version (Mobile)
    sources: [m1, b1]          # 只含 AdGuard Mobile + AdGuard Chinese
```

运行合并后，会生成 `dist/adblock-mo-lite.txt`。

### 示例 2：实验性规则

```yaml
custom_rules:
  - output: adblock-experimental.txt
    sources: [m1, m2, m3, b1]  # 含所有实验性规则
```

### 示例 3：给特定人群的规则

```yaml
custom_rules:
  - output: adblock-kids.txt
    description: FilterFusion - Family safe rules
    sources: [m1, b1]            # 较温和的规则组合
```

## 高级用法

### 引用 `bo` 类别的源

`category: bo` 的源（如 b1、b2）会被 mobile 和 pc 两条记录共享。

在 `custom_rules` 中引用时，只需填 id（如 `b1`），脚本会自动按目标类别过滤：

```yaml
custom_rules:
  - output: adblock-mo.txt
    sources: [m1, m2, m3, b1, b2]  # b1、b2 会自动使用 mobile 类别的规则
```

### 多个组合规则共存

你可以定义任意多个组合规则：

```yaml
custom_rules:
  - output: adblock-mo.txt
    sources: [m1, m2, m3, b1, b2]

  - output: adblock-mo-lite.txt
    sources: [m1, b1]

  - output: adblock-experimental.txt
    sources: [m1, m2, m3, b1, b2, x1]  # x1 是新增的源
```

每次运行合并脚本，所有组合规则都会重新生成。

## 运行合并

配置好 `custom_rules` 后，运行：

```bash
# 方式 A：统一入口（推荐）
python -m scripts.merge_all

# 方式 B：直接运行
python scripts/merge_rules.py --category mobile
python scripts/merge_rules.py --category pc
```

## 注意事项

1. **id 必须存在** — `sources` 列表中的 id 必须在上方的 `sources:` 段中定义，否则会报错
2. **去重是自动的** — 同一个规则在多个源中出现，合并时会自动去重
3. **顺序不影响结果** — 合并后的规则顺序是确定的（按类型分组），与 `sources` 中的顺序无关
4. **description 可选** — 不填会自动生成，格式为 `FilterFusion - Ad blocking rules (Category)`

## 故障排查

### 报错：`id 'xxx' not found in fetched sources`

原因：`sources` 中引用了不存在的 id。

解决：检查 `config/sources.yaml` 的 `sources:` 段，确认 id 已正确定义且未被注释。

### 生成的规则文件为空

原因：引用的源全部抓取失败。

解决：检查 `fetch_meta.json`（临时目录）中的抓取结果，确认源 URL 是否可访问。
